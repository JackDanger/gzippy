## Findings: what the derived views tell us

### F1 — The gap is per-unit-of-work, not parallelism (AMD confirms)

On AMD (frozen, bare-metal, most reliable data), native and rapidgzip have **identical
self-speedup curves** on every compute-bound corpus:

- silesia: rg S(t8)=2.17, native S(t8)=2.17 — ratio rg/native constant at 0.88-0.89 T1→T16
- model:   rg S(t8)=4.12, native S(t8)=4.13 — ratio rg/native constant at 0.84 T1→T16

If the gap were caused by bad scheduling or starvation, adding more threads would reduce
it (rg's T8 ratio would be better than T1). It is not: the ratio is **flat from T1 to T16**.
The bottleneck is in the work each unit of processing does, not how threads are coordinated.

Intel data is noisier (LXC) and shows rg scaling slightly better (model T1→T8: rg 5.1x vs
native 4.2x), but Intel is untrustworthy for this analysis due to LXC scheduling variance.
AMD bare-metal is the authoritative view.

### F2 — Small-file and IO-bound corpora are not the right bench targets

- **monorepo (49 MiB decompressed, 9 MiB compressed):** fixed startup/overhead (~150ms)
  dominates. gzippy-isal is SLOWER than rg at T4+ because the parallel overhead exceeds
  the decode savings. Not a meaningful target for decode-engine optimization.
- **storedheavy (95 MiB, stored blocks, 1.0x ratio):** IO-bound. All tools scale ~1.2x
  T1→T8 because disk write speed is the ceiling. Not a target for decode optimization.
- **bignasa (783 MiB, 75 MiB compressed):** Compute-bound by ratio (10.5x decompression)
  but the ALL-TOOL plateau at T4+ (rg: 1196-1277ms T4-T16, barely moves) suggests
  the compressed read rate is a secondary bottleneck.

The **diagnostic corpus** for engine work is silesia (202 MiB, 3.1x ratio, clean scaling)
or model (257 MiB, 1.27x ratio, excellent T1→T8 scaling). Use these for any inner-loop
or consumer-serial work measurement.

### F3 — gzippy-isal scales worse than gzippy-native on AMD

At T1: isal beats native (1.12x for silesia, 1.08x for model) — ISA-L's inner Huffman
loop is faster. At T4+: isal LOSES to native (silesia T4: isal=477ms, native=463ms) and
scales worse through T16. This points to a per-chunk overhead in the ISA-L integration
path that grows with thread count, or the ISA-L single-shot path being invoked differently.
isal_vs_native at T1=1.27 but at T8=1.01 on silesia (AMD). Something in the parallel ISA-L
path does MORE work per chunk than the pure-Rust path.

### F4 — Chunks-per-corpus: chunk count is NOT in the GZIPPY_LOG_FILE for isal build

The trace is only available from gzippy-native (GZIPPY_LOG_FILE). Confirmed via
single T4 run on silesia: 17 chunks of 4 MiB compressed each (pool_size=4,
chunk_size=4194304, total_bits=545839584). T8 uses the same 17 chunks with a larger pool.
Chunk count does NOT change with thread count — only the worker pool size changes.
