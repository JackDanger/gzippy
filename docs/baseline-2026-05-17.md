# Baseline timing — 2026-05-17

Per the verbatim user request ("get timing data"), this is the comparative
benchmark at the current branch HEAD. Committed so future perf work has
a single before/after pair, not scattered run output.

## Hardware

- `neurotic` (homelab x86_64 box, 16 logical cores). `make ship` SSHes here.
- Linux. Same machine for all tools in the table below.

## Workload

- Fixture: `benchmark_data/silesia-large.bin.gz` (162,603,323 bytes
  compressed; 503,627,776 bytes uncompressed; gzip -9; single-member).
- Operation: decompress to /dev/null.
- Threads: 16.

|---|---|---|---|
| **rapidgzip** | 1804.9 | 1.00× | vendor's compiled binary; the reference parallel decompressor |

## What this means

- gzippy currently does ~1/4 of rapidgzip's throughput on the canonical
  parallel-decompression workload.
- Of the user's four explicit original requirements ("fully correctly
  implement all of rapidgzip… zero fallback paths… get timing data…
  do it all at once, in a single change"), this commit closes #3.
- #1 and #4 are not closed. The Codex 5.5 meta-audit at
  `docs/codex-meta-audit.md` explains why: the trajectory is a museum
  of vendor-shaped ports next to a structurally unchanged production
  engine. The Rule #4 + #7 loopholes that enabled that trajectory were
  deleted in commit `f400b24`.

## Target

- "Fastest gzip ever created" (CLAUDE.md prime directive). Operationally:
  ≥ rapidgzip on this fixture, T=16. Stretch: ≥ libdeflate one-shot on
  T=1, scaled by cores.

## Where the gap lives (per the diagnostic in commit `5125422`)

Effective parallelism is T≈2 (1 consumer + 1 on-demand worker)
regardless of `parallelization=16`. Prefetched chunks reach the
BlockFetcher cache after commit `5125422`'s two-key fix, but every
prefetch task fails at the slow-path `BlockFinder` scan and falls back
to on-demand at the real offset. Slow-path scan recognizes only
dynamic-Huffman headers; real chunk boundaries land on fixed-Huffman
or stored blocks the scanner doesn't model. Closing the gap requires
mirroring vendor's `findNextDynamic + findNextUncompressed` cascade
(GzipChunk.hpp:778-794) — OR — bypassing the gzippy-original
slow-path entirely and using gzippy's SIMD inflate primitives (now
unlocked by the Rule #7 deletion in commit `f400b24`).
