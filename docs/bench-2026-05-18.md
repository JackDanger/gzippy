## Correctness bug found and fixed in this run

Two coupled bugs were uncovered when bench_all_formats.py ran on the
221 MB source (existing tests covered ≤24 MB so never hit the trigger):

### Bug 1: `ChunkData::finalize` overwrote `max_encoded_offset_bits` (b6321f7)

Vendor's `decodeChunkWithInflateWrapper` (GzipChunk.hpp:192-268) keeps
`maxEncodedOffsetInBits` at its default `numeric_limits<size_t>::max()`
for fast-path chunks. `matchesEncodedOffset(offset)` then short-circuits
to `offset == encodedOffsetInBits` (exact-match-only).

gzippy was setting `max_encoded_offset_bits = end_encoded_offset_bits`
in `finalize`, so `matches_encoded_offset(boundary)` returned TRUE for
chunk N+1's request after chunk N's end coincided with that boundary —
the speculative-trim path reused chunk N for chunk N+1's slot.

Fix: don't touch max in finalize. Construction-time default
(encoded_offset_bits) is the right value for the fast path.

### Bug 2: `decode_chunk_with_window` preempted at `max_decoded_chunk_size` (7b11b3f)

Vendor's fast path bounds output only by `exactUntilOffset` — there is
NO `alreadyDecoded >= max_decoded_chunk_size` check in
`decodeChunkWithInflateWrapper`. The preempt vendor has is in
`finishDecodeChunkWithInexactOffset` (the slow/speculative path).

gzippy's fast path preempted at 80 MiB. The preempt left the ISA-L
wrapper MID-BLOCK (no real EOB at `wrapper.tell_compressed()`). The
successor chunk's worker could not resume from that bit position →
"chunk decode failed" error. With Bug 1 in play, the consumer instead
silently exited at chunk-end, producing exactly 80 MiB of (correct)
output then erroring "output size mismatch."

Fix: remove the preempt from the fast path; mirror vendor.

The bug existed at `17fd9b2` (the cross-chunk-retry consolidation
commit) and persisted through all 14 commits of today's structural-port
work. All affected by today's commits passed the existing 24 MB test
fixture. **bench_all_formats.py on the 221 MB fixture is what caught it.**

## perf stat — per-format CPU utilization

| Format | gzippy CPUs util | rapidgzip CPUs util | gzippy IPC (core) | rapidgzip IPC (core) |
|---|---|---|---|---|
| Single-member gzip | **1.24** | 3.47 | 1.30 | 2.91 |
| Multi-member gzip | 4.13 | 6.36 | 2.86 | 1.95 |
| BGZF | 6.34 | 5.92 | 1.80 | 1.89 |

**Single-member parallelism is broken.** gzippy uses ~1 CPU on a 16-core
machine. The consumer is serialized: chunk N+1 dispatch waits for chunk
N to complete. Compare rapidgzip at 3.5 CPUs effective.

**Multi-member**: gzippy uses fewer cores (4.1 vs 6.4) but with much
higher IPC (2.86 vs 1.95) — per-cycle work is dominant, total wins.

**BGZF**: utilization parity; gzippy and rapidgzip do roughly the same
work per core. rapidgzip wins by ~12%.

Plus `*.perfstat.txt` per format (perf stat counters) and

The single-member flamegraph shows **44M page-fault samples in gzippy**
vs negligible in rapidgzip — gzippy's new no-preempt fast path
allocates buffers up to chunk_size_compressed × ratio (potentially
80+ MB per chunk), and the first-touch page-fault cost dominates the
1.24-CPU wall time. Buffer-pool pre-fault would close this.

## Open perf items (not regressions — gaps the bench newly exposes)

1. **Single-member parallelism (highest priority)**: 1.24 CPU
   utilization is the dominant gap. Consumer-side serialization is the
   prime suspect — the spec-ring pre-fill should be moved inside
   `BlockFetcher::get` (per port-reference §B5 deferred item #2), or
   the consumer should pre-dispatch ALL partition seeds upfront rather
   than one-ahead.

2. **Page-fault cost in single-member**: per the flamegraph, the
   chunk-output buffer's first-touch cost is the largest single
   consumer of gzippy's wall time. The existing buffer-recycle pool in
   `chunk_data.rs::new_with_buffers` amortizes this across chunks on
   one thread, but with a single chunk per partition the pool's reuse
   rate is low. Pre-faulting via `madvise(MADV_POPULATE_WRITE)` or
   reusing thread-pool-scoped buffers more aggressively are candidates.

3. **BGZF parity**: gzippy's BGZF path has matched CPU utilization but
   lags rapidgzip by ~12%. Flamegraph diff would show where.

## Tooling shipped this session

- `scripts/bench_stats.py` — statistically-aware bench with 95%
  bootstrap CIs on the gzippy/rapidgzip throughput ratio.
- `scripts/bench_all_formats.py` — multi-format wrapper that generates
  fixtures for every gzip-family format both tools can read, runs
  bench_stats.py per format, and (with `--profile`) captures
  flamegraphs.
- `scripts/profile_compare.sh` — perf record + inferno-flamegraph for
  gzippy + rapidgzip side-by-side.
- `scripts/off_cpu_profile.sh` — off-CPU sched-tracepoint capture to
  find blocked-thread time.
- `scripts/scaling_sweep.sh` — bench across T=1,2,4,8,16 to locate the
  parallel-scaling cliff.
- `scripts/timeline.py` / `scripts/trace_diff.py` — deterministic
  per-thread trace analysis from `GZIPPY_LOG_FILE`.

Run end-to-end (on neurotic, root):

```
python3 scripts/bench_all_formats.py \
    --source benchmark_data/logs.txt \
    --gzippy ./target/release/gzippy \
    --rapidgzip vendor/rapidgzip/librapidarchive/build/src/tools/rapidgzip \
    --bgzip /usr/bin/bgzip \
    --threads 16 --trials 20 \
    --out-dir /tmp/bench-DATE \
    --profile
```
