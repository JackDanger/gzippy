# Baseline measurements — 2026-05-19, HEAD `d32dfb5`

**Hardware**: neurotic (x86_64, 16 cores).
**Build**: `RUSTFLAGS="-C debuginfo=1 -C strip=none -C force-frame-pointers=yes" cargo build --release --features isal-compression`.
**Fixture**: `benchmark_data/silesia-large.bin` (503,627,776 bytes), gzippy-compressed at default level per `scripts/bench_all_formats.py`.
**Methodology**: 8 trials per cell, 95% bootstrap CI, interleaved trial order to cancel thermal drift (`scripts/bench_stats.py`).

This is a fact-record — actual measurements at a specific HEAD on a specific machine — not a planning doc.

T=1 multi-member and BGZF show > 100k MB/s for gzippy — bench script artifact, not real throughput. Filed as separate cleanup; doesn't change the analysis below.

### Key observations

- **gzippy WINS single-member at T=1** (1.16× vendor). Routes to ISA-L single-shot / libdeflate one-shot.
- **gzippy's single-member parallel-SM path NEGATIVELY scales**: 782 (T=1) → 459 (T=2) → 405 (T=16). rapidgzip scales the right way: 675 → 1380.
- **Multi-member and BGZF are close** (0.73× — 1.20×). Wins at T=1-2, slight losses at T≥4.
- **gzippy-parallel native format** has the same shape as SM — peaks at T=1, drops with more threads.

The SM parallel-scaling cliff is the headline defect.

## What this means for direction

- **L1 distance ≠ wall-clock perf**. The closest-to-vendor flamegraph (SM at 85 pp) is the WORST perf gap (0.29×). The most-divergent (BGZF at 115 pp) is one of the better perf ratios (0.73×). Flamegraph parity is necessary but not sufficient.
- **SM gap is the page-fault + orchestration story**. At T=16, 53% of gzippy's CPU is in those two bands; vendor: 20%. Same total CPU available, gzippy uses 35% on real decode vs vendor's 56% — and that ratio dominates wall time.
- **MM/BGZF gap is the libdeflate-vs-rapidgzip-inflate story**. A constant-factor per-thread inflate speed, not a structural deficit.

## What to do next (in priority order)

1. **Allocator experiment** — try `rpmalloc-rs` as `#[global_allocator]`. Cheapest test of "does vendor's literal allocator close the page-fault band when used globally." Branch + bench-on-branch. Per `feedback_no_innovation.md`, framed as "port vendor's literal allocator" not "swap in any arena allocator."
2. **If (1) doesn't close it** — implement Option C: `allocator-api2` polyfill + `Vec<u8, RpmallocAlloc>` for ChunkData's buffer fields only. ~200 LOC. Matches vendor's `FasterVector<u8, RpmallocAllocator>` exactly.
3. **Orchestration overhead** — chunk_fetcher::drive + parallel_gzip_reader::read_parallel_sm together account for 18% of gzippy CPU. Likely Rust trait-dispatch / closure-boxing cost. Defer until after allocator.

The MM/BGZF gap is a fixed-cost inflate-engine difference; closing it would mean swapping libdeflate for a different inflate inside the BGZF parallel pipeline. Defer until SM is closed.

  git checkout d32dfb5 && \
