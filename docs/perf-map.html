1 · What changed: the faithful memory model, completed

The clean output buffer is now rapidgzip's shape — a list of fixed 128 KiB segments (SegmentedU8, vendor DecodedData::data), markers resolved in place (no separate narrowed buffer, vendor applyWindow), with the A3 window-prefill re-added into segment 0. Prior segmented ports regressed T16 only because they had dropped A3; restoring it eliminated that. Net −619 lines (the old contiguous model + campaign oracles removed).

The gap is WAITING, not work

Instruction counts are equal (T16: HEAD uses fewer). The entire cycle gap is IPC (1.81 vs 1.98), and the IPC deficit lands in backend-memory-bound stalls (+4.3 pp at T8, +6.5 pp at T16). gzippy isn't doing more or different instructions — it's stalling on memory.

Precise locus (SM trace, T8): the in-order consumer's fetcher.get() waits ~30 ms of the ~135 ms wall on the next frontier chunk being ready, and that chunk's own decode is the memory-stall-bound region (in-place resolve cost is now ~0; apply_window ≤1.1 ms is the dominant post-step). Mechanism: the frontier chunk resolves slower per cycle because the decode is memory-stall-bound, so the in-order consumer waits. Same locus the campaign mapped — now at 1.1–1.2× rather than 1.47×.

5 · The bottom line

Segment-native A3 is a shippable memory win. It is byte-exact, cuts resident memory 23–27% toward rapidgzip's, eliminates the prior segmented port's T16 regression (by restoring the A3 window-prefill), is slightly faster at T8-file, and is −619 lines of simpler, more faithful code (it now mirrors vendor's DecodedData: 128 KiB-granule data, in-place marker resolve, front-prepend).

It does not — and is not meant to — close the speed gap. The residual 1.03–1.19× to rapidgzip is backend-memory-bound waiting on the in-order consumer's frontier-chunk decode, confirmed by TMA (equal instructions, lower IPC, higher memory-bound). That is the same architectural residual the campaign mapped and could not close with any software lever (decode is competitive; footprint is overlapped slack; coherence is kernel-side). gzippy is already the fastest decoder on arm64; this x86 residual is coordination + memory-stall, recoverable only by a deeper rearchitecture.

Honest note on how we got here: the campaign built this memory model twice and dead-ended it as "not a speed lever" — stopping one commit short of this exact fix, which its own analysis had already designed and filed as a candidate. The lesson banked: a faithful, correct, memory-positive change ships on its own merits even when it's wall-neutral.
