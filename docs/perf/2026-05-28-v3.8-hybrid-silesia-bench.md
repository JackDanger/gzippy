## Result

Both decoders **3350/3350 byte-perfect** against the predecessor-window oracle.

|-------------------------------|---------|-----|-----|-----|-----|--------|
| Pure-Rust reference (layered) | 364     | 355 | 361 | 404 | 390 | 364    |

## Verdict

**Hybrid is at parity with the highly-optimized Rust reference.** The
asm literal-decode loop is correct but does not improve throughput
because the v3.7 byte-by-byte refill is the bottleneck.

The byte-by-byte refill was introduced in commit `e642d7f` to fix a
bit-truncation bug in v3.4's 8-byte chunked refill (which used `shl
rax, cl` and dropped the top `bitsleft` bits of the load). The
correctness fix is sound; the refill cost is what kills the asm
throughput advantage.

## Next: v3.9 — restore correct chunked refill

libdeflate's canonical refill pattern:

```c
bitbuf |= load_u64(in_next) << bitsleft;  // shift load up; spilled bits LOST
in_next += (63 - bitsleft) >> 3;          // advance by actual bytes consumed
bitsleft |= 56;                           // becomes 56..63
```

The `|= 56` trick guarantees bitsleft >= 56 after refill (and < 64), so
we always have at least 56 bits available. The spilled high bits of
the load are not a bug because we advanced `in_next` by exactly the
number of bytes we *did* consume — the spilled bytes will be re-loaded
on the next refill from `in_next` + N.

This is the pattern v3.9 will adopt. Expected gain: lift hybrid from
~370 MB/s to >500 MB/s, since the v3.4 8-byte refill was the original
source of v3.4's measured throughput before the correctness bug
surfaced.

## v3.10 — in-asm match copy

Independent lever. The hybrid currently bails to Rust on every
non-literal exit, which on silesia is ~30-40% of symbols (every
match). Even if the asm refill is fast, the per-match transition
costs are high. In-asm match copy (length-base + extra bits + dist
table lookup + window copy) will land in v3.10.
