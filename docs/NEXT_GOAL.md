## 1. One-line goal

> **By end of session, the median bootstrap CI lower bound from
> `make rapidgzip-advise` on the chosen port branch reaches
> ≥ 0.60× rapidgzip on `silesia-large.gz` at T=16, up from today's
> 0.36× (cross-chunk-retry) / 0.41× (cutover-one-pr).**

0.60× is the floor. Stretch target is 0.80×. The eventual goal
(≥ 0.99× = AT PARITY) is two sessions out; setting it as the *first-
session* target is exactly the "jump 15 feet in the air" failure mode
the user named.

The metric is the **CI lower bound** from `scripts/bench_stats.py`
(N≥10 interleaved trials, 95% bootstrap CI). Median alone is
fragile — see the 0.331–0.503 spread on cross-chunk-retry today.

## 2. Branch choice — `cutover-one-pr`

This is the trunk for the next session. Reasons:

- **The user's audit verdict.** `docs/codex-meta-audit.md` §6: "Session
  3 — One commit cutover, no sub-agents… If it cannot be done in one
  commit, the answer is *we cannot satisfy your original constraint as
  stated.*" `cutover-one-pr` is that single-commit form.
- **Today's numbers are close.** 0.412× vs 0.360× — within noise but
  marginally ahead. More importantly, fewer files to reason about.
- **`feat/cross-chunk-retry` is the museum.** 82 commits, 23.6k LOC,
  most landed-unwired. The static map proves it: see §6 below.
  Keep it as **inputs** to the tooling and as a reference codebase,
  not as the trunk.

**`feat/tooling` does not merge to main.** It is rebased onto
`cutover-one-pr` and becomes part of that branch's history. Every
implementer commit on the cutover branch ships with the tooling
present. There is no "tooling separate from work" anymore.

Concrete branch action at session start:

```bash
git fetch origin
git checkout cutover-one-pr
git pull
git cherry-pick origin/feat/tooling~1 origin/feat/tooling   # bench_stats, static_map, trace_diff, advisor_loop, profile_compare, Makefile
git push -u origin cutover-one-pr      # or push to a branch if cutover-one-pr is protected
```

`feat/tooling` then deletes after the cherry-pick lands — it has no
independent reason to exist.

## 3. Workflow — the advisor loop is the harness, not a side dish

Every change in this session follows this five-step rhythm. **No commit
lands without a fresh `make rapidgzip-advise` run committed alongside
it.** This is the structural defense against the 80-commit chain.

```
1.  RUN     ssh neurotic 'cd gzippy && git pull && make rapidgzip-advise'
2.  READ    target/tooling/run-*/{symbol_map.md, trace_diff.md, bench_stats.md, next_action.prompt.md}
3.  PICK    one change from next_action.prompt.md (or paste prompt to an Opus advisor)
4.  IMPLEMENT  the change — single commit, ≤ 500 lines diff
5.  RERUN   make rapidgzip-advise → commit BOTH the change AND the new
            run-*/ artifacts in the SAME commit, message ending with
            the new CI lower bound vs the prior one.
```

If step 5 shows the CI lower bound went down or stayed within noise,
the commit reverts before push. No "hill-climb in hope" — the tool
says ship or revert.

`make rapidgzip-advise` requires:
- `RAPIDGZIP=vendor/rapidgzip/librapidarchive/build/src/tools/rapidgzip`
- `ADVISE_FIXTURE_COMPRESSED=benchmark_data/silesia-large.gz`
- `ADVISE_FIXTURE_ORIGINAL=benchmark_data/silesia-large.bin`

Note the fixture name: `silesia-large.gz`, NOT `silesia-large.bin.gz`
as in the docstring. The Makefile defaults are wrong; fix that in
the cherry-pick.

## 4. Concrete first action

Before any code change, the very first commit of the session is:

> **Cherry-pick `feat/tooling` onto `cutover-one-pr`, fix the fixture
> name in `Makefile`, run `make rapidgzip-advise`, commit the
> `target/tooling/run-<timestamp>/` artifacts as
> `docs/baseline-2026-05-18.md`.**

That gives us a tooling-driven baseline on the actual port-work
branch (today's `baseline-2026-05-17.md` was generated by hand,
not by the tooling — it cannot be re-run for comparison).

The first **code change** is then driven by `next_action.prompt.md`
from that baseline. It will almost certainly be one of:

- (A) Replace gzippy's slow-path block scanner with vendor's
  `findNextDynamic + findNextUncompressed` cascade
  (`GzipChunk.hpp:778-794`). Rationale: `baseline-2026-05-17.md`
  blames the prefetch cache for 0 hits because the scanner can't
  find fixed-Huffman starts. Rapidgzip's verbose output today
  on the same fixture shows **32 prefetch hits / 1 miss = 97%
  hit rate**. We're getting effectively T=2 parallelism; vendor
  is getting T≈12 (75–85% pool efficiency × 16 threads).
- (B) Wire one of the four "landed-unwired" symbols the static map
  flagged: the call site exists in `src/decompress/parallel/` but
  no caller in `src/` reaches it. Use `scripts/static_map.py
  --output-only-landed-unwired` (add the flag if absent).
- (C) Replace the gzippy-original `consumer_loop` (1043 LOC) call
  site with a thin `processNextChunk`-shaped trait dispatch. Per
  `codex-meta-audit.md` §4: "the sufficient gate is the CLI's hot
  path must type-check against a single trait that mirrors
  `processNextChunk`."

The advisor prompt picks one of A/B/C; the implementer does NOT
do all three.

## 5. Stop conditions

**Success — ship and end the session:**
- Bootstrap CI lower bound ≥ 0.60× rapidgzip on
  `silesia-large.gz`, T=16, N≥10 trials.
- Byte-identical output (existing `tests::routing` invariants pass).
- ≤ 5 commits total in the session (defends against the 80-commit
  chain).
- All 5 commits include their `run-*/` artifacts.

**Soft success — stop and ask:**
- 0.50× ≤ CI lower bound < 0.60×, *and* the trace_diff output names
  a single remaining bottleneck. Hand off to a human reviewer with
  the diagnostic in hand instead of trying another commit.

**Failure — stop, write a diagnostic, do NOT keep trying:**
- 5 commits land, CI lower bound still < 0.50× → the chosen
  approach is wrong. Stop. Write `docs/session-2026-05-18-failure.md`
  with the bench numbers per commit and the trace_diff verdict at
  each step. Do NOT roll into a 6th commit on a hunch.
- Any commit lands without its `run-*/` artifacts → revert the
  commit, stop the session entirely. (This is the explicit
  enforcement of rule #5 below.)
- Sub-agent dispatched → stop the session entirely. The whole
  point of this session is one implementer, one branch, one
  advisor loop. (See bias #4 below.)

**Time box:** 4 hours of wall time, or 5 commits, whichever first.

## 7. Evidence from the tooling — captured 2026-05-18, on neurotic

The tooling was cherry-picked onto a scratch branch `advise-2026-05-18`
(= `cutover-one-pr` + `feat/tooling`). Then run on
`benchmark_data/silesia-large.gz` (503.6 MB uncompressed), T=16,
N=10 interleaved trials.

| rapidgzip | 1437.7 | 1175.9 | 1915.8 | 19.62% |

### 7.2 bench_stats — feat/cross-chunk-retry `4315c2d`

|---|---|---|---|---|

The two ratios are not statistically distinguishable; the gzippy

### 7.4 rapidgzip --verbose breakdown (same fixture, T=16)

```
Time spent in block finder               : 0.017 s
Time spent decoding with custom inflate  : 3.007 s   <-- dominant
Time spent decoding with ISA-L           : 1.559 s
Time spent allocating and copying        : 0.266 s
Time spent applying the last window      : 0.184 s
Time spent computing the checksum        : 0.065 s
Non-marker symbols                       : 68.5% (345 M of 503 M)
Replaced marker symbol buffers           : 31.5% (158 M)
Pool Efficiency (Fill Factor)            : 75 – 85%
Prefetch Cache Hits / Misses             : 32 / 1   (97.4% hit rate)
Cache Hits / Misses                      : 0 / 33
```

The two numbers that should drive session 1's change:
- **Prefetch hit rate 97%** — rapidgzip's scout finds the next
  chunk boundary correctly almost every time. Gzippy's slow-path
  scanner (per `baseline-2026-05-17.md`) finds approximately *zero*
  fixed-Huffman starts — so prefetched chunks fail at the offset
  the scout guessed.
- **Custom inflate dominates wall-summed thread time.** Rapidgzip
  spends 2× more time in custom inflate than in ISA-L. Gzippy is
  inverted (uses ISA-L for everything once bootstrap succeeds).
  After fixing the scout, the inflate inner loop will become the
  next bottleneck — but it's session 2's problem.

### 7.5 trace_diff — NOT YET FUNCTIONAL

`scripts/trace_diff.py` crashed on the first run because gzippy
on `cutover-one-pr` does NOT emit JSON-lines events to
`GZIPPY_LOG_FILE` for this code path. The `trace.rs` module exists
but the marker pipeline driver doesn't call into it. **This is the
first concrete sub-task** of the first action: wire 6–10 trace
events into the cutover driver, matching the labels `trace_diff.py`
expects (`fetcher_get_start`, `block_decode_start`, `marker_resolve`,
etc. — see `scripts/trace_diff.py` lines 30–80 for the schema).

Without trace_diff, we are blind to per-phase deltas — only to
end-to-end throughput. Wiring trace is therefore the prerequisite
for **all** other changes in this session.
