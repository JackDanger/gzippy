# Phase 1.4 result — apples-to-apples Rust vs C++ on the SAME algorithm

## Setup

- Host: neurotic (i7-13700T, Debian, CPU 1 pinned via taskset)
- Compiler: g++ 13 (vendor side), rustc pinned via `rust-toolchain.toml`
- `RUSTFLAGS="-C target-cpu=x86-64-v3"` on gzippy side
- `-march=x86-64-v3` on vendor side
- Vendor built twice:
  - `build/`: `LIBRAPIDARCHIVE_WITH_ISAL=ON` (default)
  - `build_noisal/`: `LIBRAPIDARCHIVE_WITH_ISAL=OFF`, `CMAKE_CXX_FLAGS="-DWITH_MULTI_CACHED_HUFFMAN_DECODER"`
- Both vendor builds verified via `nm`: the `_noisal` binary links `rapidgzip::deflate::HuffmanCodingShortBitsMultiCached<11>` (apples-to-apples with gzippy).

## Measurements

Both tools timed on the same 30-block silesia corpus, 200 iterations.

| Variant | gzippy (Rust ShortBitsMultiCached) | vendor (C++ ShortBitsMultiCached, no ISA-L) | Ratio |
|---|---:|---:|---:|
| `inner_loop_only` (fast path) | 560.6 MiB/s | 873.7 MiB/s | **1.56×** |
| `bootstrap_path` (markers) | 484.2 MiB/s | 654.0 MiB/s | **1.35×** |

For reference, vendor WITH ISA-L on the same corpus:
| Variant | vendor (ISA-L) |
|---|---:|
| `inner_loop_only` | 837.3 MiB/s |
| `bootstrap_path` | 623.0 MiB/s |

Surprising: NoISA-L vendor is ~4% FASTER than ISA-L vendor on this corpus. Hypothesis: ISA-L's per-block table init dominates on the short blocks in this corpus; long-stream workloads should swing back to ISA-L. Per Opus advisor: 4% is within corpus noise; do not rewire production on this.

## Pivot decision (per plans/inner-loop-execution.md Phase 0.2)

**Pivot input**: bootstrap-path gap = 1.35× (apples-to-apples Rust vs C++).

**Bucket**: 1.20–1.40× → "PGO + DoubleLiteralCached only; skip Phase 2 unless those fall short."

**Opus advisor amendment**: the table assumed "variant port" meant porting a live vendor variant. DoubleLiteralCached is commented out in vendor's `deflate.hpp:182` — it's a dormant variant vendor explored and abandoned. Per CLAUDE.md "port don't innovate," skip the DoubleLiteralCached port.

**Refined action**:
1. **PGO + LTO + target-cpu=native** rebuild of gzippy.
2. Re-bench on same corpus.
3. Decision tree based on POST-PGO bootstrap ratio:
   - ≤ 1.10×: declare inner loop done; pivot to pipeline work.
   - 1.10×–1.20×: ship PGO, pivot to pipeline, log the residual.
   - \> 1.20×: port `HuffmanCodingShortBitsCachedDeflate` (vendor's live non-MULTI_CACHED default — qualifies as parity work).

## Key learnings

1. **Vendor uses ISA-L Huffman decoder by default**: this is the SOURCE of the 1.54× production gap. Apples-to-apples (no ISA-L on either side) shows only 1.35× — meaning ISA-L vs pure-Rust is ~0.19× of the gap.

2. **The 1.54× number conflated algorithm gap and toolchain gap**: gzippy's pure-Rust ShortBitsMultiCached vs vendor's ISA-L was not a fair comparison. The actionable gap is the algorithm-equal toolchain gap (1.35×), which PGO can address.

3. **DoubleLiteralCached is not vendor parity work**: it's a dormant experimental variant. Phase 4 of the original plan would have ported a non-vendor pattern.

4. **gzippy's fast path is ALREADY at vendor parity (ISA-L on both sides)** for x86_64 production via `decode_chunk_isal`. The 1.56× fast-path gap in the bench is irrelevant because the bench used pure-Rust Block for fast-path; production uses ISA-L. No optimization needed there.

5. **The 1.35× gap is BOOTSTRAP-only**. Production parallel SM spends most time in fast-path (ISA-L); bootstrap is only the first chunk + cross-chunk transitions. Real wall-time benefit of closing this is bounded by bootstrap's share of production time.
