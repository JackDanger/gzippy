## Why this exists
The chronic "#2" blocker (gz-vs-rapidgzip structural gap-finding) was that rapidgzip
only existed on the heavily-contended Intel LXC (load ~15 from other tenants → noisy).
This work puts rapidgzip + a representative corpus set on the QUIET deterministic mac,
so #2 runs instruction-exact without the Intel-contention dependency.

## D1 — rapidgzip ON THE MAC (the durable unblock)
- **Method:** `pip install rapidgzip` into a venv (`/tmp/rgvenv`). A prebuilt
  aarch64 wheel installed cleanly in seconds — no cmake build needed.
  (`python3 -m venv /tmp/rgvenv && /tmp/rgvenv/bin/pip install rapidgzip`)
- **Path:** `/tmp/rgvenv/bin/rapidgzip`  **Version:** rapidgzip 0.16.0
- **Self-test:** rapidgzip output == `gzip -d` byte-exact on all 4 corpora (sha256).
  Threading flag is `-P N` (`--decoder-parallelism`); decodes to `/dev/null` via `-c`.
- **CAVEAT (scope):** mac rapidgzip runs its **aarch64** path — there is **NO ISA-L**
  (x86-only). So this is **gz-aarch64 vs rg-aarch64**. That is VALID for the
  PIPELINE/tax structural compare (≈arch-independent) but does NOT test the x86 ISA-L
  inner-kernel gap. The Intel silesia-T4 +16% loss is gz-vs-ISA-L — that cell STAYS
  OWED to a quiet Intel window.

## D3 — THE BIFURCATION (deterministic, what DOES transfer)

work), which is ≈arch-independent. Multiplicative decomposition of the total-instr
ratio: `gz/rg = KERNEL(T1) × PIPELINE-tax`, where PIPELINE-tax = (gz T1→T inflation) /
(rg T1→T inflation).

### Structural reading (STRONG-attribution, instruction-deterministic)
- **KERNEL (T1):** gz uses 0.245–0.368× rg's single-thread instructions on aarch64
  (gz 3–4× LIGHTER). This REVERSES the Intel picture (rg has ISA-L on x86) → **does
  NOT transfer**; the x86 ISA-L inner-kernel gap is OWED to Intel.
- **PIPELINE-tax (TRANSFERS):** every T>1 cell is **CONVERGENCE** — gz's parallel
  pipeline inflates instruction-work T1→T by **1.8–4.1×** vs rg's **1.2–1.8×**, a
  **1.50–2.29× tax differential**. gz's window-absent-marker + apply-window +
  replace-markers parallel machinery emits substantially MORE coordination
  instruction-work per added thread than rg's faithful equivalent. Both tools carry
  the same machinery (gz is a port of it); rg's is the more instruction-efficient.
- **Compressibility pattern:** the tax is WORST on the small/compressible corpora
  (monorepo 1.96–2.19×, nasa 2.21–2.29×) and milder on the large/dense ones
  (silesia/squishy 1.50–1.61×). Consistent with gz's pipeline carrying a larger
  fixed per-chunk/coordination overhead that dominates more when there is less real
  decode work to amortize against (matches the HEAD commit-9df14077 "T2<T1 pipeline
  fixed-overhead" line, now extended to a gz-vs-rg differential).

### The convergence-vs-fundamental fork — ANSWER
For the gz-vs-rg PIPELINE comparison on aarch64: the fork lands on **CONVERGENCE, not
fundamental**, at every cell. gz's parallel pipeline is structurally heavier than rg's
(does more coordination instruction-work per thread); a faithful structural port of
rapidgzip's marker/apply-window pipeline can close that differential. This is a live,
arch-independent #2 convergence target — independent of the kernel (ISA-L) cells owed
to Intel.

## Owed (NOT-YET-LAW)
- **x86 ISA-L inner-kernel cells** (the T1 KERNEL ratio + the real silesia-T1/T4
  rg-WINS losses): owed to a quiet Intel window — the aarch64 kernel ratio is
  rg-no-ISA-L-contaminated and does NOT transfer.
- **AMD/Zen2 replication** for LAW (BMI2 _pext/_pdep/_bzhi microcoded on Zen2 may
  shift kernel magnitudes; the PIPELINE-tax is coordination-bound so likely holds,
  but unmeasured).
- **Gate-2 perturbation** of the pipeline-tax (e.g. an apply-window removal-oracle)
  to convert this STRONG attribution into a causal floor — not yet run.
